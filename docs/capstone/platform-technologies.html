Data Flow Diagram

This diagram shows the movement of data in the capstone project, starting from customer input, moving through the application layer, then into storage, notifications, and live queue updates.

Figure 2. Data moves from user input, into the frontend, through the backend and service layer, then into storage and outbound update channels.

How the Data Moves

  1. The customer enters booking information in the frontend interface.
  2. The frontend sends the request to the Express API.
  3. The API validates the request and applies booking, queue, and payment rules.
  4. Validated records are stored in PostgreSQL, while proof files are uploaded to Backblaze B2.
  5. The system then sends results to vendors, administrators, and customer-facing live views through notifications or SSE updates.

Blocks or Items in the Diagram

  • `Customer`, `Vendor / Staff`, and `Platform Admin` are the people who create, review, or monitor system data.
  • `Frontend Booking UI` is the point where user input is collected and turned into API requests.
  • `Express Booking API` is the controller layer that receives, checks, and responds to requests.
  • `Business Services` are the internal modules that execute booking, queue, notification, and storage logic.
  • `PostgreSQL` is the structured data store for transactions and operational records.
  • `Backblaze B2`, `Notification Providers`, and `Queue Live Updates` are outbound destinations for file storage and real-time communication.

Operational Notes

  • The monorepo uses workspace scripts from the root `package.json` to start and build the apps together.
  • The backend boots by connecting to PostgreSQL first, then starts the Express server.
  • Allowed browser origins are assembled from environment settings so the customer app, platform dashboard, and API can coexist locally and in deployment.
  • Queue updates use SSE for live public board behavior rather than WebSockets.
  • Private booking payment proofs are stored separately from public assets and are only exposed through authenticated routes.

Why This Stack Fits the MVP

React and Vite keep the customer and admin interfaces fast to develop, Express keeps the API simple to reason about, and PostgreSQL gives the project transactional consistency for bookings, queue state, and audit data.

The result is a capstone-friendly stack that is small enough to demo, but still realistic enough to support security, privacy, and architecture documentation.
